<task type="auto">
  <name>Task 1: Build fetchRoster Cloud Function</name>
  <files>functions/index.js</files>
  <action>
Add a new Cloud Function `fetchRoster` to functions/index.js that fetches a teacher's class roster from Aeries via HTTP scraping.

**Approach**: The existing `authenticateTeacher` function already successfully authenticates to Aeries via HTTP POST (form-based ASP.NET login). After login, follow the same session cookie pattern to navigate to the roster/class list pages and scrape student data.

**Implementation:**

1. **Function signature**: `fetchRoster` is an HTTPS Callable function, callable only by authenticated teachers.
   ```javascript
   exports.fetchRoster = onCall({ region: "us-central1", secrets: ["FERNET_KEY"] }, async (request) => { ... })
   ```

2. **Auth check**: Require `request.auth` — only authenticated teachers can fetch their roster.

3. **Decrypt Aeries credentials**: Read the teacher's encrypted credentials from Firestore (`teachers/{uid}/credentials/aeries`), decrypt the password using Fernet.

4. **Login to Aeries via HTTP**: Reuse the existing login approach from `validateAeriesCredentials`:
   - GET the login page, extract ASP.NET form tokens
   - POST credentials to log in
   - Capture session cookies from the response (Set-Cookie headers)
   - The redirect after successful login gives us an authenticated session

5. **Navigate to roster page**: After login, use the session cookies to make authenticated GET requests to Aeries pages that list class rosters. Aeries roster URLs typically follow patterns like:
   - `https://adn.fjuhsd.org/Aeries.net/Scheduling/StudentSchedules.aspx` (teacher view)
   - Or the teacher portal class list page
   - Use cheerio to parse the HTML response and extract student data

6. **Parse roster data**: Extract from the HTML:
   - Period number
   - Student ID (STU#)
   - Student name (Last, First)
   - Parse into the same format as current roster: `{ StudentID, LastName, FirstName }`

7. **Write to Firestore**: Write the parsed roster data to `teachers/{uid}/rosters/{period}` using admin SDK batch writes. Each period doc contains `{ roster: [...students], updatedAt: serverTimestamp(), source: 'aeries' }`.

8. **Return result**: Return `{ success: true, periods: { "1": 32, "2": 28, ... } }` with student counts per period.

9. **Error handling**:
   - If Aeries login fails: return `{ success: false, error: "Aeries login failed" }`
   - If roster page is unreachable or unparseable: return `{ success: false, error: "Could not parse roster data" }`
   - If credentials not found: return `{ success: false, error: "No Aeries credentials stored" }`

10. **Fernet decryption helper**: Add a `fernetDecrypt(ciphertext)` function (inverse of existing `fernetEncrypt`).

**IMPORTANT**: Aeries is a .NET web app. The roster page may require JavaScript execution to render. If HTTP scraping returns an empty/unusable page (common with .NET UpdatePanels or AJAX), the function should:
- Return `{ success: false, error: "roster_requires_browser", fallback: "csv_upload" }`
- This signals the client to fall back to CSV upload
- A future Playwright-based fetch on Railway (Phase 7) will handle this case

**Install cheerio** if not already in functions/package.json (check first — it was already added in Phase 5 for auth validation).

<task type="auto">
  <name>Task 2: Build roster management UI with preferred names</name>
  <files>public/index.html, "attendance v 2.5.html"</files>
  <action>
Populate the dashboard-roster section (created by Plan 01) with roster management UI.

**UI Layout:**

1. **Roster fetch controls** (top of section):
   - "Fetch Roster from Aeries" button — calls fetchRoster Cloud Function
   - Loading spinner/state during fetch
   - Success: show summary "Fetched 6 periods: P1 (32), P2 (28), ..." with a green checkmark
   - If fetchRoster returns `roster_requires_browser`: show message "Automatic roster fetch isn't available yet. Please upload a CSV roster file." with the existing CSV upload control
   - Error: show error message with details

2. **CSV upload** (keep existing functionality):
   - Move the existing roster upload UI from the Attendance tab to this Roster section
   - Keep the existing file input, upload logic, and Firestore sync
   - Label it as "Manual Upload" or "Upload CSV" as an alternative to auto-fetch
   - Remove the roster upload section from the Attendance tab (it will only live here now)

3. **Roster viewer/editor** (main content):
   - Period tabs or accordion showing each period
   - For each period, show a table/list of students:
     - Student ID
     - Official Name (LastName, FirstName — from Aeries, read-only)
     - Preferred Name (editable text input, defaults to FirstName)
     - Source indicator: "Aeries" badge for fetched students, "Manual" badge for manually added
     - Remove button (X) to remove a student from the roster
   - At the bottom of each period: "Add Student" row with inputs for ID, Last Name, First Name

4. **Preferred name data model**:
   - Add a `preferredName` field to each student object in the roster
   - When roster is fetched from Aeries or uploaded via CSV, `preferredName` defaults to `FirstName`
   - When teacher edits a preferred name, it updates the roster entry in Firestore
   - Store in roster format: `{ StudentID, LastName, FirstName, preferredName, source: 'aeries'|'manual' }`

5. **Preferred name integration throughout the app**:
   - Update the kiosk sign-in showGroup() to use `preferredName` instead of `FirstName` (e.g., "Alex, you are GROUP 3, Seat 2" where Alex is the preferred name)
   - Update the group display (both inline and fullscreen) to show preferred names
   - Update the absent list to show preferred names
   - The attendance log still records the official `Name` field (FirstName LastName) for Aeries sync compatibility, but add a `PreferredName` field to the log for display purposes
   - CSV export uses official names (LastName, FirstName from roster)

6. **Save roster changes**:
   - "Save Changes" button per period (or auto-save on change with debounce)
   - Writes updated roster (with preferred names, manual additions, removals) to Firestore `teachers/{uid}/rosters/{period}`
   - Show success/error feedback

7. **Refresh roster** (CONTEXT: manual only):
   - "Refresh Roster" button re-triggers fetchRoster
   - When refresh finds changes: show summary "2 added, 1 removed" with student names
   - Preserve manually-set preferred names when refreshing (match by StudentID)
   - Preserve manually-added students (source: 'manual') — they survive refresh

8. **Sync source HTML**: Copy public/index.html to "attendance v 2.5.html"
