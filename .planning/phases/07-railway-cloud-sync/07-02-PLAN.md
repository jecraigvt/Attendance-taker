Output: Three Python files in `railway-worker/` — worker.py (scheduler), sync_engine.py (per-teacher logic), firestore_client.py (Firebase helpers).

- No school calendar needed — skip teachers with no attendance data for today
- Only sync when NEW data exists (compare lastSyncTime vs latest attendance record timestamp)

- Invalid credentials → stop syncing for that teacher, set status to prompt credential update
- Sequential teacher processing (5-10 teachers expected)
- Per-teacher timezone stored in profile (default Pacific Time)

Create `railway-worker/sync_engine.py` — the core sync logic for a single teacher. This adapts the existing `upload_to_aeries.py` to work in the cloud context.

**Main function: `sync_teacher(uid: str) -> dict`**

Returns a result dict: `{ status, periods_processed, error, unsyncable }`

Steps:

1. **Check for new data:**
   - Call `get_last_sync_time(uid)` and `get_latest_attendance_timestamp(uid, today)`
   - If no attendance data today → return `{ status: 'skipped', reason: 'no_data' }` (don't write sync/status — nothing happened)
   - If lastSyncTime >= latest attendance timestamp → return `{ status: 'skipped', reason: 'already_synced' }` (don't write — no new data)

2. **Get credentials:**
   - Call `get_teacher_credentials(uid)`
   - If None → write_sync_status with status='failed', error='No Aeries credentials configured. Please add your credentials in Settings.' → return
   - Decrypt encryptedPassword using `cryptography.fernet.Fernet(FERNET_KEY).decrypt(token)` where FERNET_KEY is from env var
   - If decryption fails → write_sync_status with status='failed', error='Credential decryption failed. Please re-enter your password in Settings.' → return

3. **Fetch attendance data:**
   - Call `get_teacher_attendance(uid, today)`
   - If empty list → write_sync_status with status='skipped' (shouldn't happen given step 1, but guard)
   - Group rows by period (same as read_attendance_csv in upload_to_aeries.py)

4. **Playwright upload — ADAPTED from upload_to_aeries.py:**
   - `from playwright.sync_api import sync_playwright`
   - Launch headless Chromium with `slow_mo=50`
   - **Login:** Navigate to `https://adn.fjuhsd.org/Aeries.net/Login.aspx`, fill username/password, click submit, wait for redirect. Wrap in retry (3 attempts, 5s/15s/45s backoff).
     - If login fails after all retries: categorize error
       - If "Login.aspx" still in URL after submit → `credentials_invalid` → write_sync_status, return
       - If timeout/network error → `aeries_unreachable` → write_sync_status, return
   - **Navigate** to TeacherAttendance.aspx
   - **For each period** (same logic as upload_to_aeries.py):
     - Find period dropdown, select correct period
     - Click "All Remaining Students Are Present"
     - For each student: find cell by student ID using selector fallback strategy (copy SELECTOR_STRATEGIES from sync_utils.py), check/uncheck absent/tardy boxes per status
     - Track unsyncable students (student not found in grid → add to unsyncable list with reason)
     - Save after each period
   - Close browser

5. **Write sync status:**
   - On success: write_sync_status(uid, 'success', periods_processed=N, unsyncable=unsyncable_list)
   - On failure: write_sync_status(uid, 'failed', error=friendly_message)

**Error categorization:**
Create a helper `categorize_error(exception, context) -> (category, friendly_message)`:
- `credentials_invalid`: "Your Aeries password may have changed. Please update it in Settings."
- `aeries_unreachable`: "Couldn't reach Aeries. We'll retry next cycle."
- `selector_broken`: "Aeries page layout may have changed. The developer has been notified."
- `unknown`: "Sync encountered an unexpected error. We'll retry next cycle."

**Selector strategies:** Copy `SELECTOR_STRATEGIES` dict and `find_element_with_fallback()` from sync_utils.py directly into this file (or a shared utils). These are the same selectors used by the local sync — student_cell, absent_checkbox, tardy_checkbox patterns.

**Retry logic:** Implement a simple retry decorator (or inline retry loop) for the Aeries login step. Use the same backoff: 3 retries, 5s/15s/45s. Don't retry the full sync — only the login. If individual student processing fails, log it and continue to the next student (same resilience pattern as the existing upload_to_aeries.py).

**Logging:** Use Python logging module. Log at INFO level for each teacher start/end, each period processed, student counts. Log at WARNING for unsyncable students. Log at ERROR for sync failures.

Create `railway-worker/worker.py` — the main entry point for the Railway container.

**Architecture:**
- Uses APScheduler `BlockingScheduler` (simplest for a single-process worker)
- Schedules `run_all_teachers()` as an interval job every 30 minutes
- Runs on weekdays only (mon-fri)
- Active hours: 8:00 AM to 4:00 PM Pacific Time
- Graceful shutdown on SIGTERM/SIGINT (Railway sends SIGTERM on deploy/restart)

**Main components:**

1. **`run_all_teachers()`** — the scheduled job:
   - Get current time in Pacific. If outside 8:00-16:00 or weekend, log and return early.
   - Call `get_all_teacher_uids()` from firestore_client
   - For each teacher UID:
     - Log `[{uid}] Starting sync...`
     - Call `sync_teacher(uid)` from sync_engine
     - Log result: `[{uid}] Result: {status}` (success/failed/skipped with reason)
     - If exception escapes sync_teacher (shouldn't, but safety net): log error, continue to next teacher
   - Log summary: `Sync cycle complete: {N} teachers processed, {successes} success, {failures} failed, {skipped} skipped`

2. **`main()`** — entry point:
   - Configure logging: `logging.basicConfig(level=logging.INFO, format='%(asctime)s [%(name)s] %(levelname)s: %(message)s')`
   - Log startup: `Railway sync worker starting...`
   - Validate required env vars exist: `FIREBASE_SERVICE_ACCOUNT`, `FERNET_KEY`. If missing, log error and exit(1).
   - Initialize Firebase (call get_db() once to fail fast if credentials are bad)
   - Run one immediate sync cycle on startup (so deploy doesn't have to wait 30 min for first run)
   - Create `BlockingScheduler(timezone='America/Los_Angeles')`
   - Add interval job: `scheduler.add_job(run_all_teachers, 'interval', minutes=30, id='sync_job')`
   - Register signal handlers for SIGTERM/SIGINT → `scheduler.shutdown(wait=False)`
   - `scheduler.start()` (blocks forever)

3. **Signal handling:**
   - `import signal`
   - On SIGTERM: log "Received SIGTERM, shutting down gracefully...", call scheduler.shutdown()
   - On SIGINT: same behavior

4. **Health logging:**
   - Every time the scheduled job fires, log the current time and next scheduled run
   - This provides operational visibility in Railway's log viewer

**Script guard:** `if __name__ == '__main__': main()`

**Important design notes:**
- Teachers are processed SEQUENTIALLY (not parallel). With 5-10 teachers and ~2 min per sync, total cycle time is 10-20 min, well within the 30 min window.
- The worker is a long-running process, not a cron job. APScheduler runs inside the process.
- Railway keeps the container running permanently. APScheduler wakes up every 30 minutes.
- Outside school hours, the job fires but returns immediately (minimal resource usage).
