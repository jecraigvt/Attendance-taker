<objective>
Create the server-side auth infrastructure: a Firebase Cloud Function that validates Aeries credentials, encrypts them with Fernet, stores them in Firestore, and issues Firebase custom auth tokens. Also create Firestore security rules that enforce per-teacher data isolation.

Purpose: This is the security backbone of multi-tenancy. Teachers authenticate with their Aeries credentials (the only credentials they already have). 1. **Create `functions/` directory** with `package.json` containing dependencies:

   - `firebase-functions` (Cloud Function triggers)
   - `fernet` (npm package for Fernet encryption — use `fernet` package)

2. **Create `functions/index.js`** with an HTTPS callable function `authenticateTeacher`:

   Input: `{ username: string, password: string }`

   Flow:
   a. **Validate inputs** — username and password must be non-empty strings
   b. **Attempt Aeries credential validation:**
      - GET the Aeries login page to extract __VIEWSTATE, __EVENTVALIDATION, __VIEWSTATEGENERATOR tokens
      - POST the form with username + password + extracted tokens
      - Check response: if redirect (302) or response URL differs from login page, credentials are valid
      - If validation fails, return `{ success: false, error: "Invalid Aeries credentials" }`
      - If the Aeries server is unreachable or form parsing fails, proceed anyway but set a `validated: false` flag (graceful degradation)
   c. **Check if teacher already exists in Firebase Auth:**
      - Look up by email `{username}@aeries.attendance.local`
      - If exists, get the UID
      - If not, create new Firebase Auth user with that email and a random password (the real password is Aeries-managed, Firebase Auth is just for token issuance)
   d. **Encrypt Aeries password with Fernet:**
      - Read the Fernet key from `process.env.FERNET_KEY` (set via `firebase functions:config:set` or `--set-env`)
      - Encrypt the plaintext password
      - Store encrypted credentials at `teachers/{uid}/credentials`:
        ```json
        {
          "aeriesUsername": "{username}",
          "encryptedPassword": "{fernet_encrypted_string}",
          "validated": true/false,
          "updatedAt": serverTimestamp
        }
        ```
   e. **Create/update teacher profile** at `teachers/{uid}/profile`:
      ```json
      {
        "displayName": "{username}",
        "aeriesUsername": "{username}",
        "createdAt": serverTimestamp (only on create),
        "lastLogin": serverTimestamp
      }
      ```
   f. **Generate Firebase custom token** via `admin.auth().createCustomToken(uid)`
   g. **Return:** `{ success: true, token: customToken, uid: uid, displayName: username, validated: true/false }`

   Error handling:
   - Wrap entire function in try/catch
   - Return structured error objects, never expose stack traces
   - Log errors server-side with `functions.logger`

3. **Handle credential updates** — The same function handles both first-time signup and password updates. If the teacher already exists, it re-encrypts with the new password and updates the credential doc.

4. **Create `.eslintrc.js`** — basic ESLint config for Cloud Functions (or skip if not needed).

Do NOT use the `cryptography` Python library — this is Node.js. Use the `fernet` npm package which is compatible with Python's `cryptography.fernet.Fernet`. This ensures Railway (Python) in Phase 7 can decrypt what the Cloud Function encrypted.

Do NOT store the Fernet key anywhere in the codebase or in Firestore. It must come from environment config only.

  <name>Task 2: Create Firestore security rules and update firebase.json</name>
  <files>firestore.rules, firebase.json</files>
  <action>
1. **Create `firestore.rules`** at project root:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Teacher-owned data: only the authenticated teacher can read/write their own data
    match /teachers/{uid}/{document=**} {
      allow read, write: if request.auth != null && request.auth.uid == uid;
    }

    // Credentials sub-collection: extra restriction — only Cloud Functions (admin SDK) should write
    // Client can read their own credentials doc (to check if creds are set) but not the encrypted password
    // Actually: the client never needs to read credentials. Block client reads entirely.
    match /teachers/{uid}/credentials {
      allow read: if false;  // Only admin SDK (Cloud Functions) can read
      allow write: if false; // Only admin SDK (Cloud Functions) can write
    }

    // Kiosk binding: any authenticated user can read their own kiosk binding
    // Kiosk docs store which teacher a kiosk is bound to
    match /kiosks/{kioskId} {
      allow read: if true;  // Kiosks need to read their binding (teacher UID)
      allow write: if request.auth != null;  // Only authenticated teachers can bind
    }

    // LEGACY: Keep old paths readable for backward compatibility during migration
    // The kiosk still uses anonymous auth and writes to old paths until 05-03 is complete
    match /artifacts/{appId}/public/{document=**} {
      allow read, write: if true;  // Permissive during transition — tighten after Phase 5 complete
    }

    // Kiosk attendance writes: Student sign-ins go through the kiosk which may use
    // limited auth. The kiosk reads the teacher UID from its binding and writes to
    // that teacher's attendance path. This requires the kiosk to be authenticated
    // (even if anonymously) and the path must match the bound teacher.
    // NOTE: This rule is refined in Plan 05-03 when kiosk linkage is implemented.
  }
}
```

Key design decisions in the rules:
- `teachers/{uid}/{document=**}` uses recursive wildcard so ALL sub-collections (rosters, attendance, config, profile) are covered
- Credentials are locked to admin SDK only (the `allow read/write: if false` rules are overridden by admin SDK which bypasses rules)
- Legacy paths remain open during transition
- Kiosk paths allow public read so the kiosk can discover which teacher it's bound to
